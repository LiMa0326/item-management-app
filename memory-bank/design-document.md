# 家庭物品管理 App 设计文档（V0 / V1 / V2 路线图）

> 目的：为 CodeX / GitHub Copilot 提供清晰的结构、意图与迭代路径，用于 vibe coding。  
> 原则：少谈具体技术栈；优先本地计算与本地存储；云端只在“必须”时使用；不做过度设计，但为后续迭代留接口。

---

## 0. App 环境定义（目标平台与扩展策略）

### 0.1 当前目标平台
- **优先实现：Android App（手机为主）**
- 首个版本以 **本地测试与个人使用** 为主，逐步完善后再面向更多用户发布。

### 0.2 未来扩展平台（不要求在 V0/V1 实现）
- **iOS / iPhone App**：未来可能推出 iOS 客户端
- **Web 端（网站）**：未来可能提供网页端用于浏览/检索/管理

### 0.3 跨平台扩展原则（避免锁死）
> 不指定具体跨平台技术栈，只定义“可扩展的结构要求”。

- **离线优先数据模型**：核心数据结构与业务规则独立于 UI（便于 iOS/Web 复用规则与数据格式）
- **备份/同步协议稳定**：备份包格式、字段命名、版本号机制从 V0 起固定（保证跨平台一致）
- **云端接口保持通用**：未来若有云端能力（备份/同步/AI），接口与数据格式面向多客户端设计（Android/iOS/Web 可共用）
- **能力分层**：
  - 本地能力：录入、管理、搜索、导出/导入
  - 云端能力：登录、备份、同步、AI（可开关，可降级）
- **最小共同集**：Android 先行实现，但所有数据与 API 的抽象不依赖 Android 特定能力（例如避免把核心字段绑定到平台专属对象）

### 0.4 Web 端潜在定位（用于约束未来设计）
- 初期（可能的 V2+）：只读浏览 + 搜索 + 简单编辑
- 中期：支持完整 CRUD + 图片管理
- 长期：支持更强的 AI 检索、推荐与报表（可选）

## 1. 背景与目标

### 1.1 背景
家中电子产品、鞋子等物品数量较多，需要一个应用进行分类、记录、检索与维护。后续希望引入图像识别/AI 辅助录入与推荐，但首版以“可用、可扩展、成本低”为核心。

### 1.2 总目标
- **把“家里有哪些东西”变成可维护的数据**
- **尽可能离线可用**
- **未来能平滑升级到：账号、云备份/同步、AI 辅助录入与推荐**
- **未来可扩展到 iOS 与 Web（同一套数据模型/备份格式/云接口）**

---

## 2. 版本规划总览（V0 / V1 / V2）

| 版本 | 核心价值 | 是否需要云 | 主要交付 |
|---|---|---:|---|
| V0 本地 Demo | 本地可用：录入/管理/搜索/查看；先养成“记录习惯” | 否 | 本地数据 + 照片 + 搜索；可导出备份文件（跨平台可用） |
| V1 低成本云备份 + 登录（最小化） | 有账号、可把数据安全备份到云端；仍然离线优先 | 是（最小） | 登录/账户；手动“云备份/云恢复”；照片云端存储 |
| V2 同步 + AI（可选能力） | 多设备增量同步；AI 辅助录入、语义检索、推荐 | 是（按需） | 增量同步、冲突策略；AI 管线（可配置、可关闭） |

> 成本原则：V0 不用云；V1 只做“备份/恢复”而不做实时同步；V2 才引入更复杂的同步与 AI 服务。

## 3. 用户与使用场景

### 3.1 用户画像
- 个人用户（最初：仅自己）
- 未来：家庭成员、普通用户

### 3.2 典型场景
1. 买了新设备/新鞋：拍照 + 填基本信息，保存
2. 想找某物：按关键词/类别/标签快速搜索
3. 想清点：按类别浏览，查看购买时间/价格/描述
4. 想做整理：批量添加标签/移动分类/修改信息
5. 未来：扫描标签自动填名称/型号；按需求问“我该用哪个设备/鞋”
6. 未来（Web）：在电脑上快速搜索/浏览/维护库存清单（可选）

## 5. 功能需求（按版本拆分）

### 5.1 V0 功能清单（本地 Demo）
**A. 类别（Category）**
- 默认内置：电子产品（可改名、可归档、不可删除）
- 支持用户自定义类别：新增、编辑、排序、归档/隐藏

**B. 物品（Item）**
- 新增物品：基础信息 + 可选照片
- 编辑/删除（建议软删除：可恢复）
- 详情页：展示所有字段、照片墙、历史修改时间

**C. 字段（最小必需 + 可扩展）**
- 基础字段：
  - 名称（必填）
  - 购买日期（可选）
  - 购买价格（可选）
  - 购买币种 purchaseCurrency（可选，默认 USD）
  - 购买地点/网站（可选）
  - 描述（可选，长文本）
- 扩展字段（为后续迭代预留）：
  - 标签 tags[]（V0 可手动输入，V2 可 AI 生成）
  - 自定义属性 customAttributes（单字段 JSON；值类型限制为 string|number|boolean）
  - 照片 photos[]（本地 URI/路径）

**D. 浏览与检索**
- 全局搜索（名称/描述/地点/标签），规则为大小写不敏感 + 子串匹配（LIKE %kw%）
- tags 检索按整词匹配
- 按类别筛选
- 排序：最近添加/最近更新/购买日期/价格（可选，日期/价格空值放后）

**E. 备份（本地，跨平台友好）**
- 导出：生成一个备份包（建议包含数据 JSON + 照片引用/拷贝策略）
- 导入：从备份包恢复（覆盖或合并策略先做简单版：覆盖）
- 备份包格式与字段命名应保持平台无关，便于未来 iOS/Web 直接复用
- 默认策略：V0 本地导出默认 `full`；V1 云备份默认 `thumbnails`
- `replace_all` 导入策略：导入前自动创建本地快照回滚点，并清理本地照片文件（含孤儿文件）
- `checksums.json`：V0 可不生成；V1/V2 云备份默认生成并校验（SHA-256）

**V0 验收标准**
- 在无网络情况下可完整使用
- 50+ 条物品仍能流畅搜索与打开详情
- 导出后可在新安装的应用中成功导入并恢复主要信息

---

### 5.2 V1 功能清单（登录 + 最小云备份）
> 目标：**“有账号 + 数据不怕丢”**，仍然不做实时同步。

**A. 账号与会话**
- 登录/退出
- 基本账户信息（仅必要字段）
- 本地数据与账户绑定：同一设备可切换账户（切换时提示是否备份/导出）

**B. 云备份/云恢复（手动触发）**
- 一键云备份：将本地备份包上传云端
- 一键云恢复：从云端选择最新备份包下载并恢复
- 备份版本列表：显示时间、大小、包含照片数量
- 失败重试、断点续传（可选，优先简化）
- 包完整性校验：默认启用 `checksums.json`（manifest/data/photos）

**C. 成本控制策略（V1 必须遵守）**
- 默认不自动备份（由用户手动触发）
- 图片可选上传：
  - 低成本模式：仅上传压缩图/缩略图（可配置）
  - 完整模式：上传原图（可配置）
- 备份频率提示：仅提示，不强制

**D. 跨平台预留**
- 云备份包与 API 设计需保证未来 iOS/Web 可以使用同一套备份/恢复能力

**V1 验收标准**
- 登录后可成功进行云备份与云恢复
- 换设备登录后可从云恢复完整数据（至少文本字段 + 缩略图）
- 云端使用量可控：无用户操作时不产生持续费用（尽量无定时任务）

---

### 5.3 V2 功能清单（增量同步 + AI 能力）
> V2 分两条线：**同步** 与 **AI**。两条线都必须“可关闭、可降级”。

#### 5.3.1 多设备增量同步
**A. 同步模式**
- 离线优先：本地永远可用
- 增量同步：只上传变化（而不是每次全量备份）
- 同步触发：
  - 手动“立即同步”
  - 可选：应用启动/退出时尝试同步（需可开关）

**B. 冲突策略（先简单，后增强）**
- 默认策略：最后更新者覆盖（Last-Write-Wins）
- 冲突可视化（后续增强）：提示“两个设备都改了”，用户选择保留哪个

**C. 数据一致性**
- 每条记录带版本号/更新时间戳
- 删除采用软删除并同步删除标记

**D. Web/iOS 扩展支持（可选开始做客户端）**
- 同步协议与数据格式应与 Android 客户端解耦，使 iOS/Web 客户端可复用同步逻辑与后端 API

#### 5.3.2 AI / 识别能力（可选、按需启用）
**A. 辅助录入（优先本地）**
- 扫描条码/二维码（若物品有）
- OCR 读取标签（例如型号/序列号/规格）
- 图片识别：给出候选名称/类别/属性（可置信度展示）

**B. 语义检索（可选）**
- 用户输入自然语言：“那台去年买的降噪耳机”
- 返回最匹配的物品列表（可解释：匹配到的关键词/标签）

**C. 场景与推荐（可选）**
- 自动生成“适用场景”字段（例如：通勤、运动、差旅）
- 用户提问：“我周末跑步应该穿哪双？”→ 推荐鞋并给理由
- 推荐必须满足：
  - 可解释（引用物品属性/标签）
  - 可编辑（用户可修正标签与场景，形成反馈）

**V2 成本与隐私原则**
- 默认本地推断/本地处理优先（若设备能力允许）
- 如需云端 AI：
  - 必须显式开关
  - 明确告知上传内容（文本/图片）与用途
  - 允许“仅在 Wi-Fi 下”与“仅处理缩略图”模式

**V2 验收标准**
- 同一账号两台设备：新增/修改/删除可在合理时间内同步一致
- AI 入口可用：至少完成“从照片/OCR 生成候选名称并填入表单草稿”
- 关闭 AI/关闭同步后应用仍完整可用

## 6. 信息架构与界面草图（文字版）

### 6.1 导航结构（建议）
- 首页：搜索框 + 快捷筛选（类别/最近）
- 类别页：类别列表（含物品数量）
- 物品列表页：瀑布/列表视图切换（可选）

### 6.2 关键交互（新增物品）
1. 选择类别（或默认最近使用）
2. 输入名称（必填）
3. 可选：拍照/选图
4. 填购买信息与描述
5. 可选：添加标签、自定义属性
6. 保存

> V2：在第 2/3 步旁边提供“扫描/识别”按钮，识别结果写入“草稿”，由用户确认后保存。

## 7. 数据模型（抽象设计，避免绑定具体数据库）

### 7.1 核心实体

#### Category
- id (string/uuid)
- name (string)
- sortOrder (int)
- isArchived (bool)
- createdAt, updatedAt

#### Item
- id (string/uuid)
- categoryId (ref)
- name (string, required)
- purchaseDate (date?, optional)
- purchasePrice (number?, optional)
- purchaseCurrency (string?, optional)
- purchasePlace (string?, optional)
- description (string?, optional)
- tags (string[], optional)
- customAttributes (JSON object, values: string|number|bool)
- createdAt, updatedAt
- deletedAt (timestamp?, for soft delete)

#### ItemPhoto
- id
- itemId
- localUri (string)  // V0/V1 必需
- thumbnailUri (string?) // V0 起建议持久化
- contentType (string, required) // e.g. image/jpeg
- remoteUri (string?) // V1/V2 可选
- width, height (optional)
- createdAt
- 备份映射规则：`full -> <photoId>.<ext>`，`thumbnails -> <photoId>_thumb.<ext>`

### 7.2 未来扩展实体（先不实现，留接口）
- ItemUsageScenario（场景列表、来源：用户/AI）
- SyncMetadata（每条记录版本号、同步状态）
- AuditLog（可选：记录关键变更，便于冲突处理与追踪）

## 8. 本地优先与存储策略

### 8.1 本地存储（V0 起）
- 结构化数据：本地数据库（支持查询、索引、迁移）
- 图片：本地文件系统/媒体库引用
- 缓存：缩略图、搜索索引（可选）

### 8.2 备份包格式（建议）
- manifest.json（版本号、生成时间、设备信息、统计信息、exportMode）

- photos/（可选：照片文件或缩略图）
- checksums.json（V1/V2 推荐，SHA-256）

**必须包含：备份版本号**，便于未来做迁移与兼容。

## 11. 隐私与安全

- 默认离线：不上传任何数据

## 12. 性能与体验要求（不写死实现）

- 搜索响应：本地 1k 物品仍能快速返回（V0 暂不设硬阈值，记录基线；后续可升级为量化 SLA）

## 15. 已冻结决策（2026-03-03）
1. 本地导出默认 `full`，云备份默认 `thumbnails`。
2. `purchaseCurrency` 为 Item 标准字段（可选但必须建模）。
3. `customAttributes` 使用单字段 JSON，值类型限制为 `string|number|boolean`。
4. 默认分类不可删除，仅可重命名与归档；归档可在设置页恢复。
5. V0 搜索语义固定为大小写不敏感子串匹配；tags 按整词匹配。
6. V0 不强制单独回收站页；通过“显示已删除”筛选和详情页恢复入口完成恢复闭环。
7. `replace_all` 导入前必须自动快照；导入时清理本地照片与孤儿文件。
8. V0 可不生成 `checksums.json`；V1/V2 云备份默认生成并校验。
9. 1k 搜索在 V0 不设置硬阈值，先记录基线并纳入后续性能提升计划。
